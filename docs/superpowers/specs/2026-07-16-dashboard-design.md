# Streamlit Dashboard Design

**Date:** 2026-07-16
**Status:** Approved design. Implementation plan to follow.

---

## 1. Purpose

An interactive Streamlit dashboard that surfaces every output the instructions
require (Parts 2–4) plus the statistical analysis, reading the existing
`analysis/` functions live so on-screen numbers are identical to `make pipeline`
artifacts. Deployed to Streamlit Community Cloud; the public URL is the
submission's "link to the dashboard."

## 2. Decisions (from brainstorming)

- **Interactive explorer:** required outputs shown by default, with filter
  controls that re-run the same analysis functions live (cached).
- **Layout:** tabs (Overview / Response Analysis / Subset Explorer) with
  per-section inline filters; slim sidebar holds the title and a collapsible
  About/methodology panel.
- **Layered statistics:** plain-language findings callout → annotated boxplots →
  expandable full statistics table → expandable methodology.
- **Clean + light theme:** `.streamlit/config.toml` theme, `st.metric` KPI
  tiles, one consistent Plotly palette across charts, styled dataframes;
  minimal targeted CSS.

## 3. Architecture & data flow

A thin **render** layer over `analysis/` — no new analysis logic. Every analysis
call is wrapped in `@st.cache_data` keyed on filter args.

**Self-initializing database (load-bearing for deployment):** Streamlit
Community Cloud clones the repo and runs `streamlit run dashboard/app.py`; it
does not run `make pipeline`, and `cell_counts.db` is git-ignored. A cached
`get_connection()` therefore **builds the DB from the committed `cell-count.csv`
if it is missing** (via Part 1's `build_database`), then returns the connection.
The DB-build decision lives in a pure, testable `ensure_database(db_path,
csv_path)` function; the Streamlit wrapper caches it with `@st.cache_resource`
(one-time, ~1s). This makes the deployed app fully self-contained.

**File decomposition** (each file one clear responsibility):

```
dashboard/
├── __init__.py
├── app.py              # page config, theme, sidebar, tab routing
├── data.py             # ensure_database(); cached get_connection() + cached analysis wrappers
├── components.py       # filter controls, KPI tiles, dataframe styling, PALETTE, finding_summary()
└── sections/
    ├── __init__.py
    ├── overview.py     # Part 2 tab
    ├── response.py     # Part 3 tab
    └── subset.py       # Part 4 tab
.streamlit/config.toml  # theme
```

## 4. Tabs

### Overview (Part 2)
- KPI tiles: total samples, subjects, projects.
- Filters: condition, treatment, sample type (each "All" + values), population
  multiselect. Defaults to **all samples** (the required full frequency table).
- The relative-frequency table (styled `sample, total_count, population, count,
  percentage`), filtered live.
- A small mean-composition chart (mean % per population for the current filter).

### Response Analysis (Part 3)
- Cohort filters defaulting to **melanoma / miraclib / PBMC** (the required view).
- Layered stats block:
  1. **Findings callout** — plain-language summary generated by
     `finding_summary(compare_responders(...))` (e.g., cd4_t_cell candidate,
     non-concordant; or "no populations differ").
  2. **Boxplots** — `responder_boxplots(...)`, significant populations annotated.
  3. **Full statistics table** (expander) — coef, CI, p, q (mixed + MWU),
     effect size, flags; significant rows highlighted.
  4. **Methodology & assumptions** (expander) — the mixed-model / MWU / FDR /
     bootstrap rationale.

### Subset Explorer (Part 4)
- Filters: condition, treatment (baseline t=0 is fixed by definition).
- KPI tiles + breakdown charts: samples per project, responders/non-responders,
  male/female.
- The posed-question answer (**10206.15**) shown as a highlighted `st.metric`,
  computed via `avg_bcells_melanoma_male_responders_at_baseline`.

## 5. Robustness

Filters can yield empty or degenerate cohorts (one response group only, too few
subjects for the mixed model). Each tab guards these with a friendly empty-state
message (`st.info`/`st.warning`) instead of a traceback, so the explorer never
crashes on an odd filter combination. The Response tab specifically checks for
both response groups and a minimum subject count before calling
`compare_responders`.

## 6. Visual system

`.streamlit/config.toml` sets a light theme (brand primary color, background,
font). A single `PALETTE` in `components.py` (accessible, light/dark-safe;
selected using the `dataviz` skill at build time) is applied to every Plotly
chart and to the boxplots. `st.metric` for KPI tiles; a `style_dataframe` helper
formats numbers and highlights significant rows. Targeted CSS only where a native
component cannot achieve the result.

## 7. Deployment

Push to GitHub → connect the repo on Streamlit Community Cloud → it installs
`requirements.txt` and serves `dashboard/app.py`; the app self-builds the DB on
first load. The resulting `https://<app>.streamlit.app` URL goes in the README.

## 8. Testing

Streamlit UI is hard to unit-test, so logic lives in pure helpers that are tested
directly, with an app-level smoke test:

- `ensure_database(db_path, csv_path)` — unit test: builds when missing, reuses
  when present.
- `finding_summary(comparison_df, alpha)` — unit test across scenarios: a
  significant non-concordant population (candidate wording), none significant
  ("no populations differ"), a concordant hit (confirmed wording).
- `st.testing.AppTest` smoke test — runs `dashboard/app.py` against the real
  (fixture-built) DB and asserts it renders with no exception on default filters,
  exercising all three tabs. Extended to assert the Response tab renders its
  findings content.

## 9. Requirements traceability

| Requirement | Where |
|---|---|
| Display Part 2 frequency table | Overview tab (§4) |
| Part 3 boxplots + significance | Response tab (§4) |
| Part 3 statistics to support conclusions | Layered stats block + finding_summary (§4) |
| Part 4 subset breakdowns | Subset tab (§4) |
| Interactive exploration | Per-tab filters re-running analysis live (§3) |
| Public dashboard link | Streamlit Community Cloud (§7) |
| `make dashboard` target | already points at `dashboard/app.py` (Part 5) |
